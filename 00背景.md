以Cloud Foundry为代表的开源PaaS项目如日中天



PaaS提供了”应用托管“能力

- 在当时，用户普遍用法就是租一台虚拟机，然后像以前管理物理服务器那样，用脚本或者手工的方式在这些机器上部署应用。
- 部署过程中，会遇到云端虚拟机和本地环境不一致的情况。
- 所以当时云计算服务商比拼的就是谁能更好地模拟本地服务器环境。
- PaaS开源项目的出现，就是当时解决这个问题的最佳方案
- 比如Cloud Foundry，最核心的组件就是一套应用的打包和分发机制。
  - 用户把应用的可执行文件和启动脚本打进一个压缩包内，上传到云上Cloud Foundry的存储中。
  - Cloud Foundry会通过调度器选择一个可以运行这个应用的虚拟机，然后启动项目。
  - 同一个虚拟机上会启动不同用户的应用，Cloud Foundry会调用操作系统的Cgroups和Namespace机制为每一个应用做隔离。



dotCloud公司开源了容器项目：Docker



Docker的优势

- 镜像
- PaaS经常会出现本地能运行，但是在云虚拟机上需要做大量的修改和配置工作才能运行起来。
- Docker镜像解决了这个根本性的问题，它像是一个压缩包，包括了一个完整操作系统的所有文件和目录构成。



Docker本身并不能替代PaaS完成大规模部署应用的职责，所以有了其他工具。

- Docker Compose：在当个节点上编排容器
- Docker Swarm：跨节点编排容器



Google基于Borg系统推出kubernetes

- 在容器编排领域有足够的大的优势
- 在整个社区推行民主化架构



容器本身没有价值，有价值的是容器编排
